# Secret Management — Secret Reference Management

## Ground Rules

1. **Never touch plaintext Secret values** — the Agent only operates on references (paths/key names/CRDs); it does not read/generate/display plaintext
2. **Secrets must be externalized** — use Vault/External Secrets/Sealed Secrets; do not store plaintext in Git
3. **Least privilege** — Secret access follows the principle of least privilege
4. **Rotation must have a policy** — every Secret must have a rotation period
5. **Leaks must be responded to immediately** — notify a human immediately upon detecting a leak

## Process

### 1. Assess Secret Requirements

Obtain the list of Secrets to configure from `solo-to-ops.md`:
- Database passwords
- API keys (third-party services)
- TLS certificates
- Cloud provider credentials
- Service-to-service authentication tokens

### 2. Choose a Secret Management Solution

| Solution | Applicable Scenario | Advantages |
|------|---------|------|
| **External Secrets + Vault** | Enterprise-grade, dynamic secrets | Centralized management + auto-rotation |
| **External Secrets + AWS Secrets Manager** | AWS environment | Native integration |
| **Sealed Secrets** | GitOps-friendly | Encrypted, can be stored in Git |
| **SOPS** | Config file encryption | Simple and easy to use |

### 3. Generate the ExternalSecret CRD (without touching plaintext)

```yaml
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: payment-service-secrets
  namespace: production
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: vault-backend
    kind: ClusterSecretStore
  target:
    name: payment-service-secrets
    creationPolicy: Owner
  data:
  - secretKey: db-password
    remoteRef:
      key: secret/data/payment/production
      property: db-password
  - secretKey: api-key
    remoteRef:
      key: secret/data/payment/production
      property: stripe-api-key
```

**Note**: The Agent only generates "references" (secretKey + remoteRef); it does not generate plaintext values. Plaintext values are written by a human via a dedicated channel such as the Vault CLI.

### 4. Generate the SecretStore Configuration

```yaml
apiVersion: external-secrets.io/v1beta1
kind: ClusterSecretStore
metadata:
  name: vault-backend
spec:
  provider:
    vault:
      server: "https://vault.example.com"
      path: "secret"
      version: "v2"
      auth:
        kubernetes:
          mountPath: "kubernetes"
          role: "external-secrets"
```

### 5. Generate a SealedSecret (alternative)

```yaml
apiVersion: bitnami.com/v1alpha1
kind: SealedSecret
metadata:
  name: payment-service-secrets
  namespace: production
spec:
  encryptedData:
    # This is encrypted content, not plaintext
    # Generated by a human using kubeseal; the Agent only references it
    db-password: AgB8m9...encrypted content...
  template:
    metadata:
      name: payment-service-secrets
      namespace: production
```

### 6. Configure the Secret Rotation Policy

```yaml
# Vault dynamic secret (auto-rotation)
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: db-credentials
spec:
  refreshInterval: 24h  # rotate every 24 hours
  data:
  - secretKey: db-password
    remoteRef:
      key: dynamic/database/production
      property: password
```

### 7. Verify the Secret Configuration

```bash
# Verify ExternalSecret sync status
kubectl get externalsecret -n production
# Expected: STATUS=True

# Verify the Secret has been generated (do not view the content)
kubectl get secret payment-service-secrets -n production
# Expected: Secret exists

# Verify the Pod can access the Secret
kubectl exec -n production deployment/payment-service -- env | grep DB_PASSWORD
# Expected: environment variable exists (do not display the value)
```

## Prohibitions

- **Absolutely prohibited** from reading/generating/displaying plaintext Secret values
- Do not include plaintext Secrets in logs/reports/handoff documents
- Do not store plaintext Secrets in Git repositories (must be encrypted)
- Do not store Secrets in ConfigMaps (must use the Secret type)
- Do not skip the Secret rotation policy
- Do not grant the Agent permissions to read Secret values

## Security Red Lines

| Operation | Agent Permission |
|------|-----------|
| Generate ExternalSecret CRD | ✅ Allowed |
| Generate SecretStore configuration | ✅ Allowed |
| Read Secret metadata (name/type/status) | ✅ Allowed |
| Read Secret plaintext values | ❌ Prohibited |
| Generate plaintext Secret values | ❌ Prohibited |
| Modify Vault policies | ❌ Prohibited (human operation) |
| Delete Secrets | ❌ Prohibited (human operation) |
