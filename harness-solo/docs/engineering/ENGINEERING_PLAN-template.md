## 3. Shared Infrastructure

> Modules shared across multiple features. Implemented first (Phase 1) so features can depend on them. Unlike design's shared components (which can be stubbed), shared infrastructure must be functional before features start.

| Infra ID | Module | Used By Features | Implementation Priority |
|----------|--------|------------------|------------------------|
| I01 | API Client (fetch wrapper + error handling) | F01, F02, F03, F04, F05 | P0 |
| I02 | Auth Context (useUser hook + token management) | F01, F02, F03, F04, F05 | P0 |
| I03 | State Management Setup (store configuration) | F03, F04 | P0 |
| I04 | Utility Functions (formatDate, formatCurrency, etc.) | F03, F04, F05 | P1 |

**Implementation rules**:
- Shared infrastructure is implemented in Phase 1 before any feature starts
- Features must reuse shared infrastructure; re-implementing equivalent functionality is prohibited (enforced by product-engineering-review "shared module reuse" check)
- If a feature needs a new utility that other features might use, promote it to shared infrastructure rather than implementing it inline

## 4. Dependency Graph

> Feature-level dependency relationships. Used to compute topological sort for implementation order.

```
I01 (API Client) ─┬─→ F01 (Auth)
I02 (Auth Context)─┘    │
                        ├─→ F02 (Registration)
                        │
                        └─→ F03 (Dashboard) ─┬─→ F04 (Settings)
I03 (State Setup) ──────┘                     │
                                              └─→ F05 (Profile)
I04 (Utils) ────────────────────────────────────→ F03, F04, F05
```

**Dependency rules**:
- Hard dependency (must complete before): code-level import / API call / shared data model
- Soft dependency (preferred order): UI consistency / flow coherence (does not block start, but may cause rework)
- Circular dependencies are prohibited — if detected, refactor the feature decomposition

## 7. Cross-Feature Consistency Constraints

> Hard constraints checked by product-engineering-review. Each feature's verify must also respect these.

| Constraint | Rule | Check Method |
|------------|------|---------------|
| API contract consistency | Function signatures / return types / error handling consistent across features | Grep all exported functions, compare signatures |
| Dependency conflict | No version conflicts in package.json / lockfile | Read lockfile, check for duplicate packages with different versions |
| Data model consistency | Database schema / entity definitions / migration scripts do not conflict | Review all migration files + ORM entity definitions |
| Config consistency | Environment variable names / config keys / feature flag naming unified | Grep all `process.env.*` / config references, compare naming |
| Shared module reuse | Shared infrastructure (Section 3) not re-implemented per feature | Cross-reference imports, check for duplicate utility implementations |
| Integration runnability | All features combined: app starts + full test suite passes | Run app + run full test suite |
