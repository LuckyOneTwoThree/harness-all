# 工作流 G：立项引导

> 适用场景：新项目首次使用 harness-solo，需要初始化配置文件
> 核心模式：引导填写 constitution.md / SOUL.md / PROJECT.md / TECH_STACK.md

## 与其他工作流的差异

| 维度 | new-feature | **setup** |
|------|-------------|----------|
| 目标 | 实现功能 | 初始化项目配置 |
| 前置 | 无 | **install.sh 已执行** |
| LOOP | tdd→verify | **无 LOOP（配置为主）** |
| 产出 | 代码 | **配置文件填写完整** |

## 关键检查点

- [ ] SOUL.md 的 [用户名] 替换了？技术偏好填了？
- [ ] constitution.md 条款可验证？（不是"代码要好"，而是"package.json dependencies diff 审查"）
- [ ] PROJECT.md 的验收标准可测试？（不是"好用"，而是"输入X返回Y"）
- [ ] TECH_STACK.md 的测试/构建/lint 命令能跑通？
- [ ] 4 个文件都保存了？

## 失败处理

| 失败点 | 处理方式 |
|--------|---------|
| install.sh 未执行 | 提示用户先运行 install.sh |
| constitution 条款不可验证 | 帮用户改写为可验证的描述 |
| PROJECT.md 的 AC 不可测试 | 帮用户改写为可测试的描述 |
| TECH_STACK.md 命令跑不通 | 确认命令正确性，修正 |

## 与 install.sh 的分工

| 阶段 | 职责 |
|------|------|
| install.sh | 拷贝模板文件到项目目录（机械操作） |
| **setup 工作流** | 引导用户填写模板内容（智能引导） |

install.sh 只负责"文件到位"，setup 工作流负责"内容填写正确"。
